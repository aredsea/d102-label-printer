# 유비샵 확장 자동배포 시스템 구현 계획

> **For agentic workers:** 이 계획은 Phase 순서대로 실행한다. Phase 0(파일럿)이 통과해야 Phase 1+ 구현이 의미가 있다. 각 Task 끝은 독립 검증 가능한 산출물이다. 체크박스로 진행 추적.

**Goal:** 유비샵 확장의 껍데기 파일(manifest·skin.js·background 등)을 매장 PC 방문 없이 `git push`(+프로그램 vpk 1회)만으로 자동 반영해, "업데이트마다 재설치"를 영구 종식한다.

**Architecture:** 로컬 프로그램(D102LabelPrinter)이 GitHub raw와 확장 설치 폴더를 동기화하는 데몬 역할을 맡는다. 프로그램이 원격 인덱스를 폴링해 바뀐 껍데기 파일을 다운로드·교체하고, 확장 background가 변화를 감지하면 즉시 sync를 트리거한다. manifest `key`로 확장 ID를 고정해 폴더 교체와 무관하게 설정을 보존한다. loader의 로직 원격 로드는 현행 유지.

**Tech Stack:** C# .NET8(WinForms 트레이, HttpClient, System.Threading.Timer), Chrome MV3 확장(JS), Velopack(프로그램 자동업뎃), PowerShell(배포 스크립트).

## Global Constraints

- **비도메인 독립 PC** — force-install crx 원천 불가. unpacked 확장만 사용. (근거: Chrome 정책, spec §2)
- **확장 파일 쓰기는 프로그램만 가능** — 확장은 자기 폴더에 못 씀. 확장은 감지·트리거만.
- **manifest `key` 필수** — 확장 ID를 영구 고정해 `chrome.storage` 보존. key 없으면 폴더 경로 기반 ID라 매번 바뀜.
- **버전 규칙** — chrome이 새 버전을 인식하려면 manifest `version`이 올라가야 함. 증가는 [[feedback_program_version_semver]] 규칙(patch>9면 minor 올림).
- **loader 로직 로드 현행 유지** — `app-files.json`의 로직 파일은 지금처럼 push 즉시 반영. 이 계획은 껍데기 파일에만 관여.
- **부분 교체 금지** — 다운로드 실패 시 기존 폴더 유지. manifest는 항상 마지막에 교체.
- **pem**: `C:\Clauede Code\d102-label-printer\ubishop-barcode-ext.pem` (manifest key = 이 pem의 공개키). 절대 공개 금지.
- **안정 경로**: `%LocalAppData%\D102LabelExtension` (`ChromeExt.StableExtensionDir`). Chromium 브라우저가 unpacked로 가리키는 폴더.
- **다중 Chromium 브라우저**: 대부분 Chrome이나 Edge·Whale도 일부. 폴더 하나를 모든 브라우저가 공유 로드 → 동기화 데몬은 브라우저 무관하게 폴더 하나만 sync. Chrome은 무경고, Edge·Whale은 개발자모드 경고 → `ExtensionInstallAllowlist`에 `kejfp…` 추가로 억제 시도(관리자 1회, 비도메인에서도 작동 가능성, 실측 확정). 상세 spec §10.

### Task 0.1: 파일 lock 범위 측정

**목표:** chrome 실행 중 안정 경로의 각 파일을 덮어쓸 수 있는지 목록화 → sync 교체 전략 확정.

```powershell
$dir = Join-Path $env:LOCALAPPDATA "D102LabelExtension"
Get-ChildItem $dir -Recurse -File | ForEach-Object {
  try {
    $bytes = [IO.File]::ReadAllBytes($_.FullName)
    [IO.File]::WriteAllBytes($_.FullName, $bytes)   # 같은 내용 재기록
    "OK    $($_.FullName.Substring($dir.Length))"
  } catch { "LOCK  $($_.FullName.Substring($dir.Length))  → $($_.Exception.Message)" }
}
```

**검증 기준:** LOCK 파일 목록 확정 + rename 교체 가능 여부 확정. 이 목록이 Task 2.2 교체 로직의 입력.

**Files:**

**Interfaces:**
- Produces: `shell-files.json` 형식 = `{ version: "<manifest.version>", files: [{ path, sha256 }] }`. Phase 2 프로그램이 이 형식을 소비.

- [ ] **Step 1:** 껍데기 파일 집합 정의(loader 관리 파일 = `app-files.json`의 로직 파일 **제외**). 대상: `manifest.json`, `src/loader.js`, `src/localbridge.js`, `src/background.js`, `src/skin.js`, `popup/popup.html`, `popup/popup.js`, `rules/cache.json`, `icons/*`.
- [ ] **Step 2:** `build-shell-index.ps1` 작성 — 위 파일들의 SHA256을 계산해 `shell-files.json` 생성, `version`은 manifest.json의 version과 **자동 일치**:

```powershell
$root = $PSScriptRoot
$manifest = Get-Content (Join-Path $root 'manifest.json') -Raw | ConvertFrom-Json
$patterns = @('manifest.json','src/loader.js','src/localbridge.js','src/background.js','src/skin.js','popup/popup.html','popup/popup.js','rules/cache.json')
$files = @()
foreach ($rel in $patterns) {
  $full = Join-Path $root $rel
  if (Test-Path $full) {
    $hash = (Get-FileHash $full -Algorithm SHA256).Hash.ToLower()
    $files += [ordered]@{ path = $rel.Replace('\','/'); sha256 = $hash }
  }
}
foreach ($ic in Get-ChildItem (Join-Path $root 'icons') -File) {
  $hash = (Get-FileHash $ic.FullName -Algorithm SHA256).Hash.ToLower()
  $files += [ordered]@{ path = "icons/$($ic.Name)"; sha256 = $hash }
}
$out = [ordered]@{ version = $manifest.version; files = $files }
$out | ConvertTo-Json -Depth 5 | Set-Content (Join-Path $root 'shell-files.json') -Encoding UTF8
Write-Host "shell-files.json v$($manifest.version): $($files.Count) files"
```

- [ ] **Step 3:** 실행 → `shell-files.json` 생성 확인, version이 manifest와 일치 확인.
- [ ] **Step 4:** 무용지물 crx 배포 잔재 정리 — `update.xml`, `ubishop-barcode-ext.crx`는 더 이상 자동업뎃에 쓰지 않으므로 README에 "레거시(force-install용, 비도메인 미사용)"로 표기하거나 제거. (실제 삭제는 Task 4.x 전환 시)

## Phase 2 — 프로그램 sync 데몬

목표: 프로그램이 raw shell-files.json을 폴링해 바뀐 껍데기 파일을 안정 경로에 안전하게 교체.

### Task 2.1: ExtSync — 원격 인덱스 폴링 + 해시 비교 + 다운로드

**Files:**
- Create: `d102-label-printer/src/D102LabelPrinter/ExtSync.cs`

**Interfaces:**
- Produces:
  - `static Task<SyncResult> ExtSync.SyncAsync(bool force = false)` — 원격 인덱스 대조 → 변경분 다운로드 → 교체 시도. `SyncResult { bool Changed; int Updated; int Pending; string RemoteVersion; string Error; }`
  - `static string ExtSync.RawBase` = `"https://raw.githubusercontent.com/aredsea/ubishop-barcode-ext/main/"`
- Consumes: `ChromeExt.StableExtensionDir`(교체 대상), Task 0.1의 LOCK 목록(교체 전략).

- [ ] **Step 1:** 원격 `shell-files.json` 다운로드(HttpClient, `?_=<ticks>` 캐시버스트, 실패 시 조용히 반환).
- [ ] **Step 2:** 로컬 안정 경로 각 파일의 SHA256 계산 → 원격 인덱스와 비교해 **변경/누락 파일 목록** 산출.
- [ ] **Step 3:** 변경 파일을 임시 스테이징 폴더(`StableExtensionDir + ".staging"`)로 다운로드(바이트 그대로). manifest.json은 목록 마지막에 둠.
- [ ] **Step 4:** 각 스테이징 파일을 실제 경로로 원자적 교체(`File.Move(src, dst, overwrite:true)`). 교체 실패(lock)면 그 파일을 `pending` 카운트 + 스테이징에 남겨 다음 sync/종료 시 재시도.
- [ ] **Step 5:** 모든 변경이 교체되면 `Changed=true, Updated=n`. 일부 pending이면 `Pending=m`. 결과 로그.

**핵심 로직(교체 안전성):**
```
download shell-files.json → remote { version, files[] }
diff = files where local.sha256 != remote.sha256 (or local missing)
if diff empty: return { Changed:false, RemoteVersion }
mkdir staging
for f in diff (manifest.json 마지막 정렬):
    download RawBase+f.path → staging/f.path
    verify sha256 == remote (손상 방지)
for f in diff:
    try File.Move(staging/f.path → stable/f.path, overwrite) ; Updated++
    catch (IOException): Pending++   // lock — 다음 기회
return { Changed: Updated>0, Updated, Pending, RemoteVersion }
```

- [ ] **Step 6:** node/C# 단위 확인이 어려운 부분(HttpClient·파일)은 실측 대체 — 로컬에 가짜 raw(HttpListener 스텁) 또는 실제 raw로 SyncAsync 1회 호출해 diff·다운로드·교체 동작 확인.
- [ ] **Step 7:** Commit: `feat(printer): ExtSync — raw 껍데기 파일 동기화(해시 diff + 원자적 교체)`
