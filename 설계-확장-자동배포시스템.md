# 설계 — 유비샵 확장 자동배포 시스템 (재설치 종식)

작성일: 2026-07-14
관련 저장소: `ubishop-barcode-ext`(확장), `d102-label-printer`(프로그램)
상태: 설계 승인 대기 → 파일럿 검증 → 구현

---

## 1. 문제 정의

유비샵 바코드 확장은 큰 업데이트마다 **매장 PC를 직접 방문해 재설치**해야 했다. 매장·PC 수가 늘어 직접 방문이 불가능해졌고, "무조건 자동 반영"이 목표다.

재설치가 강제된 근본 원인은 세 가지가 겹친 것이다:

1. **파일이 두 종류로 갈림.** 로직 파일(config·collector·content·cache-intercept·statis)은 `loader.js`가 GitHub raw에서 매 페이지 로드 시 가져와 실행 → **push만으로 자동 반영**된다. 그러나 껍데기 파일(manifest·**skin.js**·background·loader·popup·icons)은 설치 폴더에 박혀 있어 **자동 반영 경로가 없다**.
2. **자주 바뀌는 기능이 껍데기(skin.js)에 몰려 있다.** 사이드바·다크모드·통계정렬·커서포커스·재고화·회전입고가 전부 `skin.js`(모든 페이지 대상, ISOLATED world)에 있어, 기능을 손댈 때마다 껍데기가 바뀌고 재설치로 이어진다.
3. **자동 업데이트 인프라(force-install crx)가 이 환경에서 원천 차단된다.** (아래 §2)

## 2. 검증된 제약 (근거 기반)

- **force-install(비웹스토어)은 비도메인 Windows에서 불가.** Chrome 정책상, 웹스토어 밖 확장의 강제 설치는 PC가 Active Directory / Azure AD / Chrome Enterprise Core에 가입돼 있어야만 작동한다. 매장 PC는 **독립 PC(비도메인)**이므로 정책을 정확히 써도 crx를 받지 않는다. → 이력의 *"정책은 있는데 crx 미다운로드"*, *"로직만 자동"*의 진짜 원인.
- **로컬 crx 파일 지정(external extension)도 Windows에서 차단**(Chrome 33+, 악성 확장 방지).
- 따라서 **crx 기반 자동 배포 경로는 이 환경에서 전면 불가.** 소스를 raw→로컬로 바꿔도 무의미하다.
- **loader의 원격 로직 로드는 도메인과 무관하게 완벽히 작동**한다(검증된 유일한 자동 경로).
- **manifest `key`로 확장 ID를 영구 고정**할 수 있다(공식 문서). ID가 고정되면 그 ID에 종속된 `chrome.storage`(계정·설정)가 폴더 교체와 무관하게 보존된다.
- **unpacked 확장은 실행 중 파일 덮어쓰기가 가능**하며(일부 잠긴 파일 제외), **chrome 재시작 시 폴더의 새 내용을 로드**한다.

## 3. 솔루션 — 프로그램을 "확장 동기화 데몬"으로

핵심 발상: crx 자동 업데이트가 막혀 있으니, **로컬 프로그램(D102LabelPrinter)이 GitHub와 확장 설치 폴더를 동기화하는 에이전트 역할**을 맡는다. 파일을 디스크에 쓰는 것은 프로그램만 할 수 있고(확장은 자기 폴더에 못 씀), 프로그램은 모든 매장 PC에 상주하며 Velopack 자동업뎃이 검증돼 있다.

이는 loader가 로직 파일에 하는 일을, 프로그램이 **껍데기 파일(manifest·skin·background 등)에 대해** 해 주는 것이다. loader는 페이지 컨텍스트의 휘발성 주입, 프로그램은 **디스크 폴더에 영구 기록**(chrome이 다음 시작 시 로드).

### 역할 분담

| 주체 | 역할 |
|---|---|
| **프로그램** (파일 쓰기 실행자) | 고정 폴더를 GitHub raw와 동기화 — 원격 버전 감지 → 바뀐 파일 다운로드 → 폴더 교체(잠긴 파일은 chrome 종료 후). 트레이 상주 중 주기 폴링. |
| **확장 background** (감지·트리거) | 원격 버전 변화 감지 시 프로그램에 "지금 sync" 요청(127.0.0.1) + 사용자에게 "다음 재시작 시 적용" 안내. |
| **loader**(현행 유지) | 로직 파일은 지금처럼 페이지에서 즉시 원격 로드. |

### 배포 흐름

```
git push  (manifest · skin.js · background · loader … 무엇이든)
   │
   ├─ 로직 파일           →  loader가 다음 페이지에서 즉시 반영 (현행)
   │
   └─ 껍데기 파일         →  프로그램이 raw 버전 감지 → 파일 다운로드
                            →  고정 폴더 교체 (잠긴 파일은 chrome 종료 후 원자적 교체)
                            →  다음 chrome 시작 시 자동 최신
                            →  재설치·제거·설정 소실 0
```

**핵심 이점: 확장의 모든 변경이 `git push` 하나로 배포된다.** 프로그램 재릴리스(vpk)는 이 sync 기능을 심는 **딱 한 번**만 필요하고, 이후 프로그램 C# 코드가 바뀔 때만 필요하다.

## 4. 동기화 메커니즘 (설계 수준)

- **원격 매니페스트**: 확장 저장소에 껍데기 파일 목록 + 버전 + 파일별 해시를 담은 인덱스(예: `shell-files.json`)를 둔다. 로직용 `app-files.json`과 별개.
- **프로그램 폴링**: 트레이 상주 중 주기적으로(예: 15~30분, 그리고 시작 시) 원격 인덱스를 가져와 로컬 폴더의 해시와 비교. 차이 나는 파일만 다운로드.
- **확장 트리거**: `background`가 원격 버전 변화를 먼저 감지하면 프로그램에 즉시 sync 요청(127.0.0.1)해 폴링 주기를 기다리지 않게 한다.
- **잠금 회피**: 다운로드는 임시 폴더(스테이징)에 받고, 파일별로 원자적 교체(rename)를 시도. 잠긴 파일은 chrome 종료 감지 후 교체. manifest는 마지막에 교체(부분 상태 방지).
- **버전 인식**: manifest의 `version`을 올려야 chrome이 새 버전을 인식한다. 배포 스크립트가 `version`·인덱스·해시를 함께 갱신하도록 강제한다(현행 update.xml/manifest 불일치 재발 방지).
- **오프라인/실패**: 다운로드 실패 시 기존 폴더 유지(부분 교체 금지). 다음 폴링에서 재시도.

## 5. 전환 시나리오 (이번 1회) — B안 확정 (2026-07-14)

**사용자 결정: B안 — 딱 1회 깨끗한 재설치.** 근본 구조 개선을 위해 1회 재설치를 용인.

전환 1회 절차:
1. 새 구조(고정 `key`·정책 제거·sync 데몬)로 **한 번 깨끗하게 재설치**.
2. 계정·위치 설정은 유비샵 localStorage 미러(`UB_ACCOUNTS_MIRROR_v1` 등)로 자동 복원 → 실질 손실 없음. (전환 전 미러 존재를 반드시 확인)
3. 이후 영구 자동(재설치 종식).

공통 기반 작업:
1. 무용지물 **force_installed 정책 제거**(비도메인 미작동 + unpacked ID 충돌 원인).
2. **manifest `key`로 ID 영구 고정**(이후 폴더 교체와 무관하게 storage 유지).
3. 프로그램에 **sync 데몬 기능 탑재**(vpk release 1회 → Velopack 자동 전파).

> 미채택 A안(재설치 0 / 현 ID 유지): PC별 현재 unpacked ID가 달라(폴더 경로 기반) 일괄 적용이 불확실해 제외.

## 6. 파일럿 검증 계획 (이 PC = Chrome 150, 확장·프로그램 기설치)

구현 전/후로 실측해 가정을 확정한다.

1. **파일 lock 범위**: chrome 실행 중 고정 폴더의 각 파일(특히 manifest.json, skin.js) 덮어쓰기 성공/실패 목록화.
2. **재시작 로드**: manifest `version`을 올린 폴더로 교체 → chrome 재시작 → 새 버전 로드 확인.
3. **ID 고정**: `key` 넣은 폴더 로드 → 의도한 ID로 고정되는지, storage 유지되는지.
4. **개발자모드 경고**: Chrome 150에서 시작 시 경고 팝업이 실제로 뜨는지(안 뜰 것으로 예상).
5. **정책 제거 영향**: force_installed 제거 후 unpacked 로드가 정상인지(충돌 해소 확인).

파일럿은 회사 PC 또는 이 PC 1대에서 먼저 → 통과 후 매장 전체.

## 7. 리스크 & 한계

- **즉시성**: 껍데기 변경은 **다음 chrome 재시작 시** 반영(unpacked 특성상 불가피). 매장은 매일 껐다 켜니 실질 최대 하루. 급하면 chrome 재시작.
- **신규 PC 최초 설치**: 완전 새 PC는 개발자모드 압축해제 로드 1회가 필요(비도메인+웹스토어 미게시라 불가피). 프로그램 설치 세팅에 포함되며, 이후 그 PC는 영구 자동.
- **개발자모드 경고**: 최신 크롬은 대체로 제거했으나, 뜬다면 unpacked의 감내 사항(끄기 불가한 보안 기능). 파일럿에서 확정.
- **프로그램 미실행 시**: sync가 멈춤. 그러나 프로그램은 시작프로그램 상주 + 인쇄에 필수라 항상 떠 있음.

## 8. 범위 외

- Chrome Web Store 게시(loader의 원격 코드 실행이 MV3 정책 위반이라 불가).
- skin.js 로직의 원격 로드 이관(userScripts 등) — 이 설계로 재설치가 해소되므로 불필요. 별도 과제로 남김.
- 로직 파일 배포 방식 변경(loader 현행 유지).

## 9. 성공 기준

- 확장의 껍데기 파일을 바꾼 뒤 **어떤 매장 PC도 방문하지 않고** `git push`(+필요 시 프로그램 vpk 1회)만으로 다음 chrome 재시작 시 전 매장에 반영된다.
- 그 과정에서 계정·위치 설정이 **소실되지 않는다**.
- 이번 전환 이후 **"제거 후 재로드" 요구가 영구히 사라진다**.

## 10. 다중 Chromium 브라우저 대응 (2026-07-14 추가)

매장 PC는 대부분 Chrome이나 Edge·Whale 등 Chromium 기반 브라우저도 일부 사용. 파일럿에서 Edge/Whale 사용자가 개발자모드 경고를 겪음이 확인됨.

- **폴더 공유**: 안정 경로(`%LocalAppData%\D102LabelExtension`) 하나를 모든 Chromium 브라우저가 unpacked로 공유 로드. 동기화 데몬은 브라우저 종류와 무관하게 **이 폴더 하나만 sync**. manifest `key`로 ID(`kejfpekfhpjlaifonfdlegmnoglaaphf`)도 브라우저 공통. → 다중 브라우저가 설계 근간을 바꾸지 않음.
- **최초 로드**: 브라우저별 1회(`chrome://` / `edge://` / `whale://extensions`).
- **개발자모드 경고**: Chrome은 팝업이 제거돼 무경고(실측). Edge·Whale은 뜸 → `ExtensionInstallAllowlist` 정책(Edge: `HKLM\Software\Policies\Microsoft\Edge`, Chrome: `...\Google\Chrome`)에 `kejfp…` 추가로 신뢰 표시 → 경고 억제 시도(관리자 1회, 프로그램 승격 경로 재활용). force-install과 달리 allowlist는 신뢰 표시라 **비도메인에서도 작동할 가능성** — 전환 시 실측 확정. Whale 정책 경로(Naver)는 별도 조사. 억제 실패 시 감내(기능엔 무영향).
- **storage 독립**: 각 브라우저 프로필별로 `chrome.storage`·localStorage 독립. 한 PC에서 한 브라우저 주력이면 무관. 여러 브라우저 병행 시 각각 설정(유비샵 localStorage 미러도 브라우저별).
